# 需求要点

## 核心原则

- **事件即标签** - 时间块是原子单位，事件只是贴在时间上的标签，不可形成层级或分组结构
- **数据即事实** - 记录原始数据，不赋予好坏、进步、退步等主观意义
- **呈现而非分析** - 用可视化工具展示发生了什么，不解读为什么
- **过犹不及** - 一切功能皆有边界，不因可能性而预设，不因扩展性而过度设计

---

## 功能边界

### 要
- 支持倒计时（默认35/90分钟，可自定义任意时长）
- 支持正向计时
- 支持暂停/继续
- 支持为时间块标记事件名称（可选）
- 完整记录每次会话的元数据：起始时间、结束时间、总时长、暂停次数、累计暂停时长、关联事件
- 支持历史数据的可视化呈现（如热力图、柱状图等）
- 数据本地持久化存储
- 支持数据导出（JSON/CSV）

### 不要
- 不支持任务列表、待办事项、项目管理等层级结构
- 不支持打卡机制
- 不支持成就、等级、连续天数等激励或惩罚
- 不支持自动分析、趋势解读、智能建议
- 不预设"番茄工作法"等固定概念框架（35/90分钟仅是默认值，非强制）
- 不因"未来可能"而预先设计复杂的数据模型或架构

---

## 数据记录参考（仅供参考）

每次时间会话应记录：

| 字段 | 说明 |
|------|------|
| id | 唯一标识 |
| start_time | 开始时间戳 |
| end_time | 结束时间戳 |
| duration | 实际持续时长（秒） |
| pause_count | 暂停次数 |
| pause_duration | 累计暂停时长（秒） |
| event_name | 关联事件名称（可选） |
| timer_type | 倒计时 / 正向计时 |

---

## 边界约束

- 事件名称不预设枚举，按需创建，不强制填写
- 可视化形式不预设穷举，按需扩展，不强制开发
- 数据结构满足当前存储需求，不过度抽象
- 不因"万一以后要同步"而预设云端字段或预留架构
